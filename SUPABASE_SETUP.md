# Настройка Supabase и админ-панели блога

Статьи блога хранятся в таблице `posts` в Supabase. Публичные страницы
`/blog` и `/blog/[slug]` читают только опубликованные статьи, а
админ-панель `/admin` (защищена входом по почте и паролю) позволяет их
создавать, редактировать и удалять.

## 1. Создайте проект в Supabase

1. Зайдите на [supabase.com](https://supabase.com) и создайте новый проект
   (можно на бесплатном тарифе).
2. Выберите регион (например, ближайший к вашим пользователям) и задайте
   пароль базы данных — он не связан с паролем админки сайта, сохраните его
   отдельно на будущее.
3. Подождите 1–2 минуты, пока проект инициализируется.

## 2. Скопируйте ключи API

В панели проекта откройте **Project Settings → API** и скопируйте:

- **Project URL** (вида `https://xxxxxxxx.supabase.co`)
- **anon public** key (длинная строка в разделе "Project API keys")

## 3. Добавьте переменные окружения

В корне проекта создайте файл `.env.local` (он уже в `.gitignore`, в git не
попадёт) на основе `.env.example`:

```bash
NEXT_PUBLIC_SUPABASE_URL=https://xxxxxxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=вставьте-anon-public-key-сюда
```

Больше никаких ключей (в частности, **service role key**) сайту не нужно —
всё разграничение прав идёт через Row Level Security (RLS) в базе.

## 4. Создайте таблицу и загрузите демо-статьи

1. В панели Supabase откройте **SQL Editor → New query**.
2. Откройте файл [`supabase/schema.sql`](./supabase/schema.sql) из этого
   репозитория, скопируйте всё содержимое и вставьте в редактор.
3. Нажмите **Run**.

Скрипт создаст таблицу `posts`, настроит политики безопасности (RLS) и
добавит 6 демо-статей, которые уже были на сайте. Скрипт безопасно
перезапускать повторно — таблица и политики создаются с `if not exists` /
`drop ... if exists`, а демо-статьи не задвоятся благодаря
`on conflict (slug) do nothing`.

### Если таблица posts уже создана раньше

После внедрения Tiptap-редактора нужно один раз выполнить миграцию
[`supabase/migrate-tiptap.sql`](./supabase/migrate-tiptap.sql):

1. Supabase → **SQL Editor → New query**
2. Вставьте содержимое `migrate-tiptap.sql`
3. Нажмите **Run**

Это добавит колонки `content_json`, `content_html`, `tags`, `seo_title`,
`seo_description` и создаст bucket `article-images` для загрузки картинок.
Без этого шага блог/админка могут писать в консоль
`column posts.content_json does not exist`.

### Если обложка не загружается («Bucket not found»)

В Supabase ещё нет Storage-bucket для картинок. Выполните отдельно
[`supabase/setup-storage.sql`](./supabase/setup-storage.sql):

1. Supabase → **SQL Editor → New query**
2. Вставьте содержимое `setup-storage.sql`
3. Нажмите **Run**

После этого перезагрузите админку и снова загрузите обложку.
Либо создайте bucket вручную: **Storage → New bucket** → имя
`article-images` → включите **Public bucket**, а политики всё равно
нужно взять из `setup-storage.sql`.

## 5. Понизьте минимальную длину пароля (важно!)

По умолчанию Supabase Auth требует пароль **не короче 6 символов**. Пароль
`admin`, который вы просили использовать, состоит из 5 символов — без этого
шага создать такого пользователя не получится.

1. Откройте **Authentication → Sign In / Providers** (в некоторых версиях
   интерфейса: **Authentication → Providers → Email** или
   **Authentication → Policies**).
2. Найдите настройку **Minimum password length** и поставьте `4` (или
   меньше).
3. Сохраните изменения.

> Это ослабляет общие требования к паролям в проекте. Если в этом Supabase
> проекте кроме админки сайта будут и другие пользователи — задайте им пароли
> длиннее и не понижайте это ограничение сильнее, чем нужно.

## 6. Создайте пользователя-администратора

1. Откройте **Authentication → Users → Add user → Create new user**.
2. Email: `admin@mail.ru`
3. Password: `admin`
4. Включите переключатель **Auto Confirm User** (чтобы не требовалось
   подтверждение почты) и сохраните.

Это единственный пользователь, у которого есть доступ к `/admin` — вход
проверяется напрямую в Supabase Auth.

## 7. Установите зависимости и запустите проект

```bash
npm install
npm run dev
```

Откройте [http://localhost:3000/admin/login](http://localhost:3000/admin/login),
войдите под `admin@mail.ru` / `admin` — откроется панель управления статьями.

## Как пользоваться админкой

- **`/admin`** — список всех статей (включая черновики), кнопки
  «Редактировать» и «Удалить».
- **`/admin/new`** — форма создания статьи: заголовок, URL (slug —
  подставляется автоматически из заголовка, можно поправить вручную),
  категория, краткое описание, обложка (необязательно, ссылка на картинку),
  текст статьи и переключатель «Опубликовать».
- **`/admin/[id]/edit`** — редактирование существующей статьи с теми же
  полями.
- Текст статьи поддерживает тот же упрощённый Markdown, что и раньше:
  `## заголовок`, `### подзаголовок`, `- пункт списка`, `**жирный текст**`,
  `[текст](ссылка)`, `---` для разделителя.
- Время чтения рассчитывается автоматически по количеству слов в тексте.
- Если снять галочку «Опубликовать» — статья сохранится как черновик и не
  будет видна на публичном сайте, но останется в списке в `/admin` для
  дальнейшего редактирования.

## Безопасность

- В браузере и на сервере используется только публичный **anon key** —
  создание/редактирование/удаление статей разрешено в базе только для
  авторизованных пользователей (политики RLS в `supabase/schema.sql`),
  анонимные посетители сайта могут только читать опубликованные статьи.
- Пароль `admin` очень простой и подходит только для демо/личного проекта.
  Если сайт будет публичным, смените пароль в
  **Authentication → Users → (нужный пользователь) → Reset password** на
  более надёжный.
